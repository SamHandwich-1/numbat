 * List of tool names that are auto-allowed without prompting for permission.
 * These tools will execute automatically without asking the user for approval.

Confirmed: field name is `allowedTools`, type `string[]`. JSDoc is unambiguous on its role — it's the auto-approval list, NOT the surface restriction. ### `tools` — restriction of the available tool surface (sdk.d.ts:1258–1267)

Confirmed: field name is `tools`, accepts `string[]` (and a preset variant we don't use). This is the actual surface-restriction mechanism, per JSDoc on `allowedTools`.

### `disallowedTools` — denylist trump card (sdk.d.ts:1226–1231)

```typescript

 * List of tool names that are disallowed. These tools will be removed
 * from the model's context and cannot be used, even if they would
 * otherwise be allowed.
 */

```

Confirmed: field name is `disallowedTools`, type `string[]`. JSDoc: "cannot be used, even if they would otherwise be allowed" — this is the trump card against any settings layer (`managed`, `flag`, `local`, `project`, `user`) that might re-introduce a tool.

### `permissionMode` — six documented values (sdk.d.ts:1521–1529 + sdk.d.ts:1862–1865)

JSDoc on the field declaration (sdk.d.ts:1521–1529):

```typescript
/**
 * Permission mode for the session.
 * - `'default'` - Standard permission behavior, prompts for dangerous operations
 * - `'acceptEdits'` - Auto-accept file edit operations
 * - `'bypassPermissions'` - Bypass all permission checks (requires `allowDangerouslySkipPermissions`)
 * - `'plan'` - Planning mode, no execution of tools
 * - `'dontAsk'` - Don't prompt for permissions, deny if not pre-approved
 */
permissionMode?: PermissionMode;
```

JSDoc on the type itself (sdk.d.ts:1862–1865) — adds documentation for `'auto'` which the field declaration omits:

```typescript
/**
 * Permission mode for controlling how tool executions are handled.
 * 'default' - Standard behavior, prompts for dangerous operations.
 * 'acceptEdits' - Auto-accept file edit operations.
 * 'bypassPermissions' - Bypass all permission checks (requires allowDangerouslySkipPermissions).
 * 'plan' - Planning mode, no actual tool execution.
 * 'dontAsk' - Don't prompt for permissions, deny if not pre-approved.
 * 'auto' - Use a model classifier to approve/deny permission prompts.
 */
export declare type PermissionMode = 'default' | 'acceptEdits' | 'bypassPermissions' | 'plan' | 'dontAsk' | 'auto';
```

All six values now documented in the installed types. No empirical test needed — the JSDoc is authoritative for what each value means.

### Per-value analysis against our requirement

Requirement: with `tools: [Read, Edit, Write, MultiEdit, Grep, Glob]` restricting the surface and `allowedTools: [same six]` pre-approving, can all six execute without producing a permission prompt that would hang the detached worker?

| Value | Behaviour | Verdict for our case |
| --- | --- | --- |
| `'default'` | Prompts for dangerous operations. | Read/Grep/Glob are non-dangerous → no prompt; Edit/Write/MultiEdit are dangerous → prompt. **HANGS the worker on edits.** |
| `'acceptEdits'` | Auto-accepts file edit operations. | Edit/Write/MultiEdit auto-accepted. Read/Grep/Glob behaviour not explicit in JSDoc — likely no prompt (read-only operations aren't "dangerous"), but not guaranteed by the installed types. |
| `'bypassPermissions'` | Bypasses all permission checks; requires `allowDangerouslySkipPermissions: true`. | Defeats the entire restriction system. This is Option A. |
| `'plan'` | No execution of tools. | Agent can't do work. |
| `'dontAsk'` | Don't prompt for permissions, deny if not pre-approved. | Six tools in `allowedTools` = pre-approved = execute. Anything NOT in `allowedTools` = denied silently (not prompted). **Worker cannot hang on a permission prompt.** |
| `'auto'` | Use a model classifier to approve/deny permission prompts. | Delegates each decision to a model call. Non-deterministic, slower, less suitable for a tightly-scoped worker. |

**Conclusion on permissionMode**: `'dontAsk'` is the cleanest match. It guarantees no prompt path exists at all — every reachable tool is either pre-approved-by-name (executes) or silently denied (no prompt). `'acceptEdits'` would also work in practice but its read-only behaviour is not explicitly documented in the installed JSDoc, so picking it would be the same kind of "infer-from-name" move §0a exists to avoid.

### Is `permissionMode` load-bearing here?

With `tools: [the six]` already restricting what the model can emit AND `allowedTools: [the same six]` auto-approving them, every reachable tool is pre-approved by name. Per the JSDoc on `allowedTools`: "These tools will execute automatically without asking the user for approval." So in *this exact config*, no tool reachable by the model would trigger a permission prompt regardless of `permissionMode`.

Where `permissionMode` becomes load-bearing:

1. **Drift defence.** If a future change adds a tool to `tools` and forgets `allowedTools`, `'dontAsk'` denies the new tool silently. With `'default'`, that same drift would prompt → hang the worker.
2. **Settings-layer re-introduction.** Settings layers (`managed` / `flag` / `local` / `project` / `user`) may add tools at runtime. `'dontAsk'` keeps the deny-by-default posture intact against those.
3. **Belt-and-braces semantics.** The combination `tools` (surface) + `allowedTools` (auto-approve) + `disallowedTools` (deny trump card) + `permissionMode: 'dontAsk'` (no-prompt mode) is four independent layers all pointing at the same outcome. Any single one being misconfigured doesn't open a hang/escape path.

So permissionMode is not *strictly necessary* for the current config but is *required for safety against drift*. It stays.

### Is `disallowedTools` redundant?

With `tools: [the six]` already excluding Bash, the model cannot emit Bash because Bash isn't on its menu — so for the model-emission path, `disallowedTools: ['Bash', ...]` is redundant.

However, the JSDoc on `disallowedTools` says: "These tools will be removed from the model's context and cannot be used, **even if they would otherwise be allowed**." The "even if otherwise allowed" clause is the operative one — `disallowedTools` is the final say across all settings layers. If a managed/project/user settings layer re-introduces Bash to the available toolset at runtime, `disallowedTools` overrides it.

So `disallowedTools` is **belt-and-braces redundant for the current static config** but **load-bearing against any settings-layer leak**. Same drift-defence reasoning as `permissionMode: 'dontAsk'`. Keeping it.

## Adapted permission config (re-presented, fully .d.ts-backed)

```typescript
// lib/feathertail/permissions.ts — every field backed by an installed-.d.ts quote.
// See docs/sdk-audit-2026-05-16.md for the JSDoc and signatures.

export function getPermissionConfig() {
  return {
    // sdk.d.ts:1258–1267 — restricts the available tool surface to these six.
    // The model literally cannot emit Bash/WebFetch/WebSearch because they
    // don't exist in its toolset.
    tools: ['Read', 'Edit', 'Write', 'MultiEdit', 'Grep', 'Glob'],

    // sdk.d.ts:1205–1211 — pre-approves these tools so they execute without
    // a permission prompt. Every tool reachable via `tools` above is also
    // in this list, so no prompt path exists for any reachable tool.
    allowedTools: ['Read', 'Edit', 'Write', 'MultiEdit', 'Grep', 'Glob'],

    // sdk.d.ts:1226–1231 — trump-card denylist. "Cannot be used, even if they
    // would otherwise be allowed." Redundant against `tools` for the static
    // config but defence-in-depth against any settings-layer drift that
    // re-introduces Bash et al. at runtime.
    disallowedTools: ['Bash', 'WebFetch', 'WebSearch'],

    // sdk.d.ts:1521–1529 + 1862–1865 — 'dontAsk' = "Don't prompt for
    // permissions, deny if not pre-approved." Combined with `allowedTools`
    // above, the six file tools execute and anything else is silently
    // denied. No prompt path can hang the detached worker.
    permissionMode: 'dontAsk' as const,

    // NOT set:
    // - allowDangerouslySkipPermissions (would defeat the whole config).
    // - canUseTool callback (not needed; the static config is sufficient).
  };
}
```

### Why `'dontAsk'` over `'acceptEdits'`

Both work for the current config. `'dontAsk'` is preferred because:
- Its JSDoc explicitly documents the "deny if not pre-approved" semantics for *every* tool, not just edits.
- `'acceptEdits'` documents auto-acceptance of edits but is silent on read-only tools; reading "default behaviour applies" into the JSDoc would be exactly the inferential move §0a is designed to catch.
- `'dontAsk'` is symmetric with the way we've configured `allowedTools` (pre-approval list). The two work together as a single explicit policy: "approved or denied; never prompted."
