### §1 · Original plan position

[`0013-slice-6-plan.md`](0013-slice-6-plan.md) §4 sub-slice 6c sketched the deliverable as `lib/debrief/opus-debrief.ts` exporting `generateDebrief(sessionId: string, messages: SDKMessage[])` — two arguments, with `db` implicitly defaulting to `sbAdmin`. The function would pull project bundle + skills + spec + priorDebrief via `ContextLoader.buildFor(projectId, 'session', sessionId)`, call Opus through `callOpusObject` with the 6a-shipped `DirectDebriefSchema`, write the `llm_calls` row first via the 6b-widened writer, then write the `debriefs` row with `debrief_type = 'direct'` and `llm_call_id` referencing the call. A standalone CLI `scripts/generate-debrief.ts` with a `--messages-file <path.jsonl>` flag would replay against a captured stream. No worker integration (that's 6d's deliverable). Vitest unit tests for Zod validation, prompt template snapshot, and DB row shape.

The proposal stage refined three load-bearing details before any code landed (the gate's job, before the Gate 1 file drafts):

- **Diff provenance.** The proposal initially left "where does the diff come from" implicit. `ContextLoader.buildFor('session', …)` returns `SessionContext` (verified at [`lib/orchestration/context.ts:26-30`](../../lib/orchestration/context.ts)) — which does **not** include `task` or `diff`. Plan 0013 §3.1 explicitly says: "the session's `task` and `diff` come from the session row directly and are passed alongside the context object, not through the loader." Resolved as: `generateDebrief` does its own `select('project_id, task, diff').eq('id', sessionId).maybeSingle()` on the sessions row. For the CLI replay path, the diff is **persisted** on the row by Slice 4's `captureDiff` — so no `--diff-file` CLI flag is needed; `--messages-file` is the only ephemeral input (messages are consumed in-memory by the worker per pre-flight Item 2, never persisted).
- **`new_concept` field-name reconciliation.** Plan 0013 §3.1 sketched `new_concept: { name, definition }`, but the codebase ships `{ title, body }` per [`lib/types/jsonb.ts:211-214`](../../lib/types/jsonb.ts)'s `DebriefNewConcept` and the production page render at [`app/sessions/[sessionId]/page.tsx:152-156`](../../app/sessions/%5BsessionId%5D/page.tsx). Resolved per 6a §2's "existing convention wins" — `lib/debrief/opus-debrief.ts` imports `DirectDebriefSchema` from [`lib/types/debrief.ts`](../../lib/types/debrief.ts) (which re-exports `DebriefContent` from `jsonb.ts`), never re-sketches the schema, and the prompt instruction text in [`lib/llm/prompts/opus-debrief.ts:43`](../../lib/llm/prompts/opus-debrief.ts) uses the live field names.
- **Method name on the ContextLoader.** Public API at [`lib/orchestration/context.ts:79-109`](../../lib/orchestration/context.ts) is `buildFor(projectId, scope, secondaryId)`. `loadAndAssertSession` (line 115) is a private method that `buildFor` calls internally — 6a renamed it from `assertSessionInProject` and widened it to return `{ specId }`. 6c's call site uses `buildFor('session', sessionId)` verbatim.

The **timeout** was reconciled by `git blame -L 75,75 CLAUDE.md` returning `aeb55bf9 / James / 2026-05-27 / "Opus debrief 90s"` — the 6b Gate 2 commit had already updated the Resilience line from 60s to 90s. `timeoutMs: 90_000` stands on the record, not assertion.

### §3 · Chosen shape

**`generateDebrief(db, sessionId, messages)`** — three-argument signature, not the two-argument sketch in plan 0013 §4 sub-slice 6c. The `db` parameter is required and goes first. Three concrete reasons:

- Matches the existing DI seam. `ContextLoader(db)` constructor takes `db` first; `insertLlmCallFromAiSdkResult(db, input)` takes `db` first. 6c's generator joins the same shape — one rule for "this function uses Supabase," not three.
- Keeps `lib/debrief/opus-debrief.ts` free of a top-level `import { sbAdmin } from "@/lib/supabase/server"`. `server.ts` reads env vars at module load and throws on miss; making any module that wants a default `db` depend on env-load discipline at import time is brittle. The caller (worker, CLI, test) decides when env is ready.
- Test isolation. Mocked-DB unit tests pass a `makeMockDb()` client directly; no need to stub `sbAdmin` at module-mock level. The nine unit tests in `lib/debrief/opus-debrief.test.ts` use this seam — 6d's worker and `scripts/generate-debrief.ts` pass `sbAdmin` explicitly.

**Plan 0013 §4 sub-slice 6c's two-argument sketch is superseded.** 6d's worker invocation is written against the three-argument contract (`generateDebrief(sbAdmin, sessionId, messages)`); the deviation is recorded here so 6d's author doesn't read the plan and miss the change.

**Discriminated-union return.** `GenerateDebriefResult` is `{ ok: true; debriefId; llmCallId } | { ok: false; llmCallId; errorKind; message }`. The five-value `GenerateDebriefErrorKind` is `OpusCallErrorKind | 'session_not_found' | 'db_read_failed' | 'context_load_failed' | 'db_write_failed'`. Per CLAUDE.md "Errors: typed error returns over thrown exceptions for expected failures" — every branch in the function returns the discriminated quadruple, no throws escape. Worker callers in 6d treat `ok: false` as a soft fail and continue to `transitionToAwaitingReview` regardless (plan §3.1's "fails open").

**Write order: llm_calls → debriefs.** Plan §3.1 spelled out the contract: PostgREST doesn't do cross-table transactions, so the `llm_calls` row is written first (returns `{ id }` via the 6b-widened writer), then the `debriefs` row carries `llm_call_id` pointing back. If the second write fails after the first succeeds, the orphan `llm_calls` row is harmless — still an accurate accounting record. The strict-order assertion is locked at `lib/debrief/opus-debrief.test.ts` test (1) via Vitest's `invocationCallOrder`.

**Two-part prompt structure with byte-stable prefix.** The stable prefix carries only project-scoped data (system instruction + CLAUDE.md + active specs as `id — goal` summary + skills + the last 30 decisions as `created_at · type · id` summary). The dynamic suffix carries session-scoped data (task, optional spec body for the session's own spec, optional prior debrief, diff, message transcript, closing instruction). Hash is computed on the suffix only by `callOpusObject` (6b Gate 3 test 4 locks this). Slice 7's four Bilby stages will reuse the same prefix shape — four cache hits per plan run on the shared project bundle is the architectural payoff.

### §4 · Rejected alternatives

- **`generateDebrief(sessionId, messages)` with default `db = sbAdmin` via a top-level import.** Considered at the proposal stage to match plan 0013 §4 sub-slice 6c verbatim. **Rejected** because the implicit `sbAdmin` import forces `server.ts` evaluation (and therefore env-var loading) at module load, even in test contexts that pass their own client. The three-argument `(db, sessionId, messages)` shape matches `ContextLoader(db)` and `insertLlmCallFromAiSdkResult(db, input)`, keeps the file pure of env-load dependency, and is what 6d's worker uses anyway. Small caller-side deviation from the plan sketch; large gain in module hygiene.
- **A separate writer (`insertLlmCallErrorFromAiSdkResult` or inline `from('llm_calls').insert(...)`) for the failure path.** Considered when the proposal first flagged "if `insertLlmCallFromAiSdkResult` doesn't accept `error`, 6c needs a sibling writer or an inline insert." **Rejected** because the pre-commit read of [`lib/supabase/llm-calls.ts:181-194`](../../lib/supabase/llm-calls.ts) found `error?: LlmCallErrorT | null` already on `InsertLlmCallFromAiSdkResultInput` (since 6b Gate 4) — the 6b suite simply didn't exercise that branch. One writer for the AI-SDK path is the contract; 6c honours it via `writeErrorLlmCall` which delegates to `insertLlmCallFromAiSdkResult` with `error: { message, subtype: errorKind }` populated and `usage: ZERO_USAGE`. The test 9 added to `llm-calls.test.ts` is the lock.
- **`as LanguageModelUsage` on the `ZERO_USAGE` placeholder.** The first draft used the cast as a convenience. **Rejected** at the commit gate because (a) the cast was masking three genuinely missing fields per the real shape at `ai/dist/index.d.ts:267-325` (`inputTokenDetails.noCacheTokens`, `outputTokenDetails: { textTokens, reasoningTokens }`, `totalTokens`), and (b) it violates CLAUDE.md "Coding conventions" — no `any`, no narrowing casts; structs the wrapper controls don't qualify for the 0016 §6 (1) typed-runtime-guaranteed-boundary cast exception. Replaced with a fully-conforming literal typed `const ZERO_USAGE: LanguageModelUsage = { … }` — every required field present, every value 0 (not undefined). Same fix applied to `fakeOpusResult.usage` in the test suite. The de-cast is what makes the typecheck gate meaningful (§6 (1)).
- **Keep the `if (!("sessionId" in ctx))` runtime guard after typing `ctx: SessionContext`.** Considered as belt-and-braces defence. **Rejected** as dead code — once `ctx` is explicitly typed `SessionContext` (the explicit type was required because `ReturnType<ContextLoader["buildFor"]>` resolves to the *last* declared overload, `PlanContext`, not `SessionContext`), the guard's branch is unreachable. Duplicate enforcement creates false comfort and forces the next reader to mentally simulate a path TypeScript already guarantees can't happen. The `buildFor('session', sessionId) → SessionContext` overload signature is trusted at this call site; 6h's live exercise is the integration backstop. Conscious reduction in runtime defensiveness, not a silent deletion (§6 (3) records the pattern).

### §5 · Findings folded in before commit

The Gate 1 file drafts landed clean at typecheck-zero on the second `pnpm typecheck` run; the first run surfaced three issues that the commit gate caught before push. Each finding has a dedicated test or runtime guard locking the corrected behaviour.

**(a) `LanguageModelUsage` real shape vs. assumed shape.** The first `ZERO_USAGE` draft was `{ inputTokens: 0, outputTokens: 0, inputTokenDetails: { cacheReadTokens: 0, cacheWriteTokens: 0 } } as LanguageModelUsage`. The real shape at [`node_modules/ai/dist/index.d.ts:267-325`](../../node_modules/ai/dist/index.d.ts) requires `inputTokenDetails.noCacheTokens`, `outputTokenDetails: { textTokens, reasoningTokens }`, and `totalTokens` (all `number | undefined`). The cast was masking three genuinely missing fields; the de-cast surfaced them and the literal was expanded to conform. The writer's `computeCostUsd` only reads `inputTokens / outputTokens / inputTokenDetails.cacheReadTokens / inputTokenDetails.cacheWriteTokens` (verified at [`lib/supabase/llm-calls.ts:79-95`](../../lib/supabase/llm-calls.ts)), so the audit-row cost is 0 either way — but the typecheck gate now means something. Same expansion applied to `fakeOpusResult.usage` in [`lib/debrief/opus-debrief.test.ts`](../../lib/debrief/opus-debrief.test.ts) so `mockResolvedValueOnce(fakeOpusResult)` typechecks against `OpusObjectCallResult<DirectDebriefT>`.

**(b) Mock-arg index typing.** Two test assertions used `mockedInsertLlmCall.mock.calls[0]?.[0]` to inspect what they thought was the input. But `insertLlmCallFromAiSdkResult` takes `(db, input)` — so `[0]` is the `SupabaseClient` argument (the actual bug position) and `[1]` is the input object the assertions actually wanted. The typed `vi.mocked(insertLlmCallFromAiSdkResult)` spy surfaced the wrong-arg cast as a TypeScript "conversion may be a mistake" error (paraphrased — the cast target `{ duration_ms: number }` does not overlap with the spy's first-arg type `SupabaseClient<Database> | undefined`). Untyped spies like the local `debriefInsertSpy = vi.fn()` would not have surfaced the same bug because their `calls` shape is `unknown[]`, so a cast off any index passes silently. Fix moved both assertions from `[0]` (db arg) to `[1]` (input object), the position they meant to read all along. Worth noting as a near-miss — the typed-spy index check caught what the test assertions did not, and is a small case for preferring `vi.mocked(typedFunction)` over plain `vi.fn()` when the function under spy is typed.

**(c) `ReturnType` on an overloaded method.** The first draft typed `ctx` as `Awaited<ReturnType<ContextLoader["buildFor"]>>` to defer to the inferred return shape. TypeScript resolves overload method types to the *last* declared signature; `buildFor`'s last overload is `(projectId, "plan", planId): Promise<PlanContext>`, so `ctx` got typed `PlanContext` and the subsequent reads (`ctx.spec`, `ctx.priorDebrief`) failed with TS2339. Fixed by importing `type SessionContext` from `lib/orchestration/context.ts` and typing `ctx: SessionContext` directly — the assignment from `buildFor(projectId, "session", sessionId)` then uses the second overload's `Promise<SessionContext>` return type and the read paths typecheck. The dead `if (!("sessionId" in ctx))` guard was removed at the same time (§4 above).

**(d) The `sessionErr` branch mislabel.** The pre-Fix-2 code returned `errorKind: "session_not_found"` from both the `sessionErr` (DB error) and `!session` (no-row) branches. But `.maybeSingle()` populates `error` only on a real DB failure — a no-row hit returns `{ data: null, error: null }`. Conflating "DB read failed" with "session doesn't exist" loses fidelity that 6d depends on (a transient DB error and a missing row warrant different recovery paths). Added `"db_read_failed"` to `GenerateDebriefErrorKind` (symmetric with the existing `"db_write_failed"`) and routed the `sessionErr` branch to it; kept `"session_not_found"` for the `!session` case. Test (9) in `lib/debrief/opus-debrief.test.ts` locks the new branch — mocks a `sessionError`, asserts no Opus call, no `llm_calls` write, no `debriefs` write. The widening was free: the union is consumed only by the as-yet-unwritten 6d worker. Rationale at length:

> `.maybeSingle()` returns `{ error }` only on a real DB failure (transient connection, RLS denial, etc.); a no-row hit returns `data:null` with `error:null`. A `sessionErr` is therefore a read failure, not a missing row — surface it as the symmetric counterpart of `db_write_failed`. The two error kinds stay distinct because the recovery paths differ: a transient DB error during the session SELECT can retry (a future slice may add reconciliation); a missing session is a permanent state and the worker should not attempt the debrief at all.

### §6 · Convention established (1) + patterns observed (2, 3)

One refinement of an existing convention earned codification this sub-slice; two further patterns are observed at one instance each and recorded here as candidates for codification on recurrence, per the recurrence-before-codification rule.

**(1) De-cast over conform — refines the 0016 §6 (1) typed-runtime-guaranteed-boundary-cast convention.** The 0016 convention permits a single `as` cast at boundaries where a strict-typed surface meets a runtime-validated one (the AI SDK's `generateObject` returning `output<TSchema>` vs. the wrapper exposing `z.infer<TSchema>` is the canonical case). 6c's Fix 1 surfaced the inverse: when a struct **literal** that this code controls needs a cast to satisfy a type, the first move is to make the literal fully conform — not to expand the cast. The boundary in (1) is between two surfaces neither side can fully describe to the other; a literal we author is *not* such a boundary. Concretely: `ZERO_USAGE` and `fakeOpusResult.usage` were authored locally, the cast was hiding three genuinely missing fields, and removing the cast surfaced the gap. The 0016 (1) exception stays in place for the SDK boundaries it was named for; 6c's refinement is: don't use the exception for cases where you control both sides. **This is codified — refining an existing convention with a clearly-bounded scope addition counts as the same convention; no new instance needed.**

**(2) Delegate over duplicate at the writer boundary — pattern observed, one instance.** When a code path needs to insert a row that another module already owns the writer for, the path delegates rather than running its own `from(table).insert(...)` chain. `writeErrorLlmCall` in `lib/debrief/opus-debrief.ts` calls `insertLlmCallFromAiSdkResult(db, { …, error: { … } })` rather than running an inline insert. The contract was endorsed at proposal stage ("one writer for the AI SDK path"); recording it here keeps the option open for codification once a second instance surfaces. Tactical consequence already visible: when a writer's input shape doesn't cover a new use case (e.g. the `error` parameter was already on `InsertLlmCallFromAiSdkResultInput` since 6b Gate 4 but untested), the right move is to test the existing surface and use it — not to write a sibling helper. The 6b Gate 4 + 6c test 9 pair are the lock. **Not yet a codified convention** — single instance (the 6b/6c writer pair); the recurrence-before-codification rule applies. A second consumer of the same shape (e.g. Slice 7's Bilby stages writing through the same widened writer, or a future debrief regeneration script reusing the same error-row path) would earn the codification at that point.

**(3) Trust the contract over re-check — pattern observed, one instance.** When TypeScript's overload typing makes a runtime branch unreachable, deleting the runtime guard is correct. 6c's `ctx: SessionContext` typing made the `if (!("sessionId" in ctx))` guard dead code; removing it was a conscious reduction in runtime defensiveness, not a silent deletion. The pattern: trust the contract at the call site once the typing makes the alternative branch structurally impossible; the integration test layer (here, 6h's live exercise) is the backstop, not duplicate runtime checks. The corollary worth recording: if a runtime guard's branch is structurally unreachable, its presence creates false comfort — the next reader thinks "oh, there's a check for that" when actually nothing can reach the guard. Better to delete and rely on the contract. **Not yet a codified convention** — single instance (the ctx-guard removal); a second case of "TypeScript overload narrowed the alternative branch to unreachable, dead runtime guard deleted" would earn the codification.

### §7 · Near-term scheduled cleanup — loader-ordering fix

Not actually a V2 question — a scheduled fix with a named trigger and a forward-pointer slot already reserved at `0019-…`.

**The gap.** During the commit-gate review, the prefix purity check on `buildStablePrefix` was clean (no clock reads, no session-scoped data) — but the production cache-stability of the prefix also depends on `ContextLoader.buildFor('session', …)` returning `specs`, `skills`, and `recentDecisions` in a deterministic order across calls. Verified at the time:

- [`lib/orchestration/context.ts:260-273`](../../lib/orchestration/context.ts) `loadRecentDecisions` — `.order("created_at", { ascending: false }).limit(30)` ✅ deterministic.
- [`lib/supabase/queries/skills.ts:27-28`](../../lib/supabase/queries/skills.ts) `listSkillsForProject` — `.order("usage_count", { ascending: false }).order("name", { ascending: true })` ✅ deterministic given data (note: `usage_count` mutations are a content-change cache bust, same shape as decision writes; the order itself is stable).
- [`lib/orchestration/context.ts:278-289`](../../lib/orchestration/context.ts) `loadSpecs` — `.from("specs").select("*").eq("project_id", projectId)` ❌ **no ORDER BY**.

Postgres makes no row-order guarantee without `ORDER BY`. Two consecutive `ContextLoader.buildFor('session', …)` calls within the Anthropic 5-min TTL window can return specs in different orders → prefix bytes differ → cache miss. The 6c snapshot fixture pre-orders its specs array, so the snapshot test can't catch this; only a live two-call exercise during 6h would surface it (and the 6b §10 watchpoint on the 4096-token cache-payload caveat would surface the same symptom, making attribution harder).

**Trigger condition.** This fix MUST land before 6h's first live cache observation — the hit-rate measurement isn't meaningful against a non-deterministic substrate. Scope: one-line change to `loadSpecs` adding `.order('created_at', { ascending: false })` (matching the convention `loadRecentDecisions` already uses). Filed as "6a gap surfaced by 6c," not a 6c slip — `loadSpecs` was the same shape at 6a close-out and the snapshot pattern would not have caught it then either. Lands as its own commit + own decisions entry (`0019-…`) per the convention from 0017 §6 (2) ("mid-slice side-tasks get their own commit + decision log, never get bundled into the host sub-slice's commits").

The lesson worth carrying forward (not yet a codified convention — single instance): prefix-builder consumers add an ORDER BY check to their close-out's verification list. Any cache-stable prefix builder is only as deterministic as its weakest input ordering. A second prefix-builder slice (Slice 7's Bilby stages share the same prefix shape) would earn the codification.
