## Slice 5 close-out — operator action surface + session lifecycle

> **Date:** 24 May 2026.
> **Type:** slice close-out.
> **Subject:** Slice 5 (status-aware ActionBar; `decisions.session_id` and `decisions.plan_id` FK behaviour as SET NULL with payload-snapshot preservation; `dismissed_at` soft-hide column + UI; dismiss/undismiss as decision types; `deriveSessionAffordances` as the shared seam between detail-page and list affordances) verified end-to-end against the ten-item manual checklist in 0009 §"Manual verification criterion". Five carry-forward items recorded for V2 or later slices; two of the five are pattern codifications (operational this slice), three are deferred work with explicit trigger conditions.

---

### What shipped, by sequencing step

Slice 5 ran as five named steps against the plan in [`0009-slice-5-operator-action-surface-session-lifecycle.md`](0009-slice-5-operator-action-surface-session-lifecycle.md), with the step-0a schema audit returning a STAY WHOLE verdict (the 5a/5b prepared-retreat option from 0009 §"Considered plan" never needed to fire). Step 4 split into 4a/4b at sign-off — a planned split along a natural seam (backend before UI) rather than a mid-stream pivot. Each step landed via the same gate pattern: draft → hold for review → implementation → hold for diff review → commit + push.

**Step 1** ([`6c5c54b`](https://github.com/SamHandwich-1/numbat)) — FK behaviour change on `decisions.session_id` and `decisions.plan_id` from `NO ACTION` to `ON DELETE SET NULL` via migration 0007, plus the `sessions.dismissed_at timestamptz` column, plus the Zod-schema extension on `DecisionPayload`'s four in-use variants (approve, redirect, kill, start_work) carrying optional `session_label` and `plan_label` snapshot fields. The snapshot mechanism rides inside the existing `decisions.payload jsonb` (Step 0a §A's finding) rather than as new columns; the schema-side migration is two ALTER statements for the FKs and one ALTER for the column. Two repeatable live-DB tests in `lib/supabase/decisions-fk-set-null.test.ts` formalise the manual probes that recovered the step from its mid-step divergence (see *Migration-and-type-sync* in §Carried forward).

**Step 2** ([`22af10e`](https://github.com/SamHandwich-1/numbat)) — ratified the existing `sessions.current_step` behaviour (KEEP populated through terminal transitions) and documented it as a Numbat convention in [`0010-current-step-on-terminal-transitions.md`](0010-current-step-on-terminal-transitions.md). The decision is "ratify current behaviour" rather than change it; five pointer comments at the relevant mutation sites (`transitionToBlocked`, `transitionToKilled`, `transitionToKilledDirectly`, `recordDecision`'s approve and kill branches) plus a column-type comment in `db.ts` make the semantic discoverable from inside the code rather than requiring a grep into the decisions log. No behavioural change.

**Step 3** ([`71ef6d4`](https://github.com/SamHandwich-1/numbat)) — `lib/orchestration/affordances.ts` introducing `deriveSessionAffordances(session): SessionAffordances` as the shared seam between the detail-page ActionBar (spine 1) and the list dismiss UI (spine 3). Pure function over `(status, dismissed_at)` returning a five-key boolean record (`approve`, `redirect`, `kill`, `dismiss`, `undismiss`). Sixteen-row table test covering every `status × dismissed_at` combination plus a coverage-completeness assertion (see *Exhaustive-enum-coverage* in §Carried forward).

**Step 4a** ([`1bd78ca`](https://github.com/SamHandwich-1/numbat)) — extended the decision pipeline backend for the dismiss/undismiss types. Migration 0008 widened the `decisions.type` check constraint; `DecisionPayload`'s discriminated union gained two new variants; `recordDecision` gained two new branches (with guards mirroring the affordances helper's rules, kept in lock-step via SYNC comments on both files); `ReviewDecisionType` renamed to `OperatorDecisionType` to match the broadened scope (the original "Review" prefix accurately described approve/redirect/kill but broke on dismiss/undismiss); `insertSessionFixture` extended with an optional `dismissed_at` arg to enable the already-dismissed test setups without ceremony.

**Step 4b** ([`f1dfdcf`](https://github.com/SamHandwich-1/numbat)) — UI rewiring. The detail-page ActionBar moved out of its awaiting_review-only mount into the page footer, gated on `shouldMountActionBar(affordances)`; this closes the 0006 carry-list gap where stuck idle/running rows had no kill affordance. SessionCard split into server shell + client `DismissButton` island; the button sits top-right beside the timestamp on terminal rows (or dismissed rows, with the un-dismiss variant). New `ShowDismissedToggle` lifts the default `dismissed_at IS NULL` list filter via the `?dismissed=show` URL param. A small `KNOWN_STATUSES` typing fix lets `?status=killing` URLs resolve (the Slice 4 transient status was silently rejected before). Snapshot capture (`payload.session_label = session.slice_name`) added to every approve/redirect/kill POST in the rewritten ActionBar, closing the step-1 snapshot contract end-to-end.

### Manual verification

Verified by the operator against the ten-item checklist from the step-4b implementation hold (the consolidated form of 0009 §"Manual verification criterion" plus regression checks). **All ten items passed.** Items 6 and 10 are the load-bearing gaps this slice closed:

- **Item 6** — opening a non-terminal, non-dismissed session detail page (e.g., `running`) now mounts the ActionBar at the page footer showing only Kill. Pre-Slice-5 these rows had no UI affordance to retry or kill, which forced the operator to direct-DB-write a kill in Slice 4's recovery (Run #2 of [`0006`](0006-slice-4-close-out.md)). That direct-write path is no longer required.
- **Item 10** — `/sessions?status=killing` now returns rows with the transient `'killing'` status. Pre-Slice-5 the page silently rejected the URL because `KNOWN_STATUSES` was missing the value (a Slice-4-introduced status that never got added to the URL filter set).

The eight regression items confirmed the existing flows survived: default-filter, toggle behaviour, dismiss/un-dismiss round-trip, awaiting_review ActionBar, terminal-row TerminalBanner, direct-URL access to dismissed sessions. Manual-run-before-close-out is the rule codified in 0006's productive-failure note; Slice 5 is the first slice to apply the rule end-to-end as a routine final gate rather than catching defects mid-stream.

### Carried forward

Five items collected through steps 1–4. Two are pattern codifications (operational from this slice forward); three are deferred work with explicit trigger conditions.

**1. Atomic-write Postgres function for V2 audit integrity.** `recordDecision` writes the decision row and the session UPDATE non-atomically — a V1 concession inherited from `create-session.ts` (decision is canonical; session UPDATE retries on failure). Slice 5 extended the same concession to the dismiss/undismiss branches. For dismiss specifically the cost of drift is low (the operator can re-click; the row stays in its prior state). As the decisions log becomes the V2 LLM-router's training data, audit-log integrity matters more — a failed UPDATE after a successful INSERT means the log claims an action the session state doesn't reflect. **Trigger for action:** any V2 work that consumes the decisions log as training data, or any observed instance of decision/session drift in production. **Likely fix:** a Postgres function (or transaction wrapper inside `recordDecision`) doing both writes atomically. Defer; not a Slice 5 blocker.

**2. `recordDecision` file-size split-point.** Step 4a added ~131 lines to `lib/supabase/mutations/decisions.ts` (two new guards + two new side-effect branches + an expanded SYNC comment + the renamed type). The function now handles five decision types via a per-type `if`/`else if` chain. **Trigger for action:** the next decision type added to the file (likely a Bilby flow in Slice 6+ — `accept_critique`, `reject_critique`, `ship`, or `edit_spec` getting their first write site). **Likely fix:** per-type handler functions or a small registry (`Record<OperatorDecisionType, (db, session, payload) => Promise<void>>`) replacing the chain. Not a refactor for refactoring's sake; the chain is fine for five types and the registry is fine for ten — trigger is when the chain length crosses the threshold where reading the function becomes scanning rather than understanding.

**3. Component-rendering test infrastructure trigger.** Slice 5's UI work shipped with pure-unit tests on affordances-driven render branching (`shouldMountActionBar`, `shouldMountDismissButton`) and no React Testing Library coverage on the actual components. The decision is documented at step 4b sign-off and codified here: V1 UI correctness is covered by manual verification per [`0006`](0006-slice-4-close-out.md)'s productive-failure pattern. **Trigger for action:** component count grows past ~10 OR a regression slips through manual QA. The current `components/` count is well under that ceiling; the manual checklist held this slice. When either condition fires, introducing `@testing-library/react` as its own slice (with vitest's jsdom env) is the move — not a mid-slice scope expansion.

**4. Migration-and-type-sync discipline.** Pattern codified across this slice. Migration 0007 (step 1) added the `sessions.dismissed_at` column but didn't update the hand-maintained `Session` type in `lib/types/db.ts`; the gap surfaced in step 3 when the affordances helper became the first reader and triggered a typecheck failure. Step 4a applied the discipline preemptively — migration 0008's `decisions.type` widening came in the same commit as the `DecisionType` union extension in `db.ts`. **The rule:** until `supabase gen types typescript` is wired in, every migration touching schema must also update `lib/types/db.ts` in the same commit. **The structural fix** (the auto-gen migration) is on the Slice 5 carry-list inherited from Slice 4 (Supabase CLI bump from v1.226.4 to v2.x); the gap closes when that lands. Until then, the rule holds. This is a *pattern codification* — already operational, not a TODO.

**5. Exhaustive-enum-coverage test pattern.** The 17th assertion in `lib/orchestration/affordances.test.ts` asserts that the 16-row table covers every `SessionStatus × dismissed_at` combination. Adding a new `SessionStatus` value to the schema breaks the test until the table is extended — drift caught at the right moment. **The rule:** anywhere a helper enumerates rules per enum value, add a table-completeness assertion alongside the per-row coverage. Cheap to write, catches the additive-enum-but-forgot-the-handler failure mode. **The pattern is operational** from step 3 forward; codified here for visibility to future contributors. Specific reusability target: the next Slice 6+ helper that branches on `DecisionType` or `PlanStatus`.

### Calibration takeaway

Three patterns repeated cleanly across Slice 5; one new pattern introduced; one process-correction logged for next time.

**Repeated cleanly.** (1) The gate pattern (draft → hold → implement → diff review → commit + push) ran at every step boundary, including the 4a/4b split, and caught issues at the planning stage rather than mid-implementation — the Step 0a schema audit's STAY WHOLE verdict was the most consequential gate output (it determined slice shape) and was data-driven rather than vibe-driven. (2) The "preserve in-flight observation data unless preservation would actively mislead" rule from [`0010`](0010-current-step-on-terminal-transitions.md) generalised across the slice — the `decisions.session_id` SET NULL + snapshot pattern in step 1 and the `current_step` ratification in step 2 are two instances of the same rule applied at different scales. (3) The "manual verification before close-out" rule from [`0006`](0006-slice-4-close-out.md)'s productive-failure note ran as a formal final gate rather than catching defects mid-stream; the ten-item checklist passed end-to-end.

**New pattern.** The 4a/4b split-by-natural-seam (backend before UI) shipped as planned with no mid-stream revision. The split criterion was reviewability (13–15 file changes get skimmed; 7–8 each get read) plus dependency direction (UI binds against a stable contract). Worth recording as the default approach when a single step crosses ~10 file edits: split at the type/contract boundary, ship the backend with tests, then ship the UI against the stable interface. The cost of two commits is small; the cost of a 13-file commit that reviewers skim is structural.

**Process correction.** Step 1 had a mid-step divergence between framework state (`schema_migrations` table) and the actual DB — the operator ran statement 1 of migration 0007 directly in the SQL editor before `pnpm db:push` ran statements 2 and 3, then I marked `0007` applied in the framework before probing the live FK state. The probe revealed only statement 1 had landed; recovery took the *roll-forward* path (operator ran statements 2 + 3 in the editor; framework consensus already correct). **The lesson:** probe-first ordering. The behavioural probe (insert + delete + observe) is empirical and reversible; marking a migration applied is administrative and propagates assumptions. When live-DB state is uncertain, probe before reconciling the framework. Recorded as informal practice; no rule change.
